# spawn-engine

`spawn-engine` is the integration loop: the crate that turns the standalone library crates into a runnable engine. It owns the `App` configuration aggregate and the `Engine` runtime, and it defines the one frame that ties everything together — the platform run loop drains into input, the asset server's loaded payloads are applied, a fixed-step accumulator advances simulation independently of the display rate, the variable- and fixed-rate ECS schedules run, render-relevant state is extracted into backend-owned proxies at a single sync point, a render backend turns those proxies into a presented frame, and the audio engine is pumped. Nothing below it knows about the loop; this crate is where platform, input, ECS, physics, render, asset, and audio become a single tick. It is domain-agnostic — it wires subsystems and owns timing, but holds no game logic, which arrives as systems, fixed hooks, and extraction routines an application registers on the `App`.

## Design Decisions

**A fixed-step accumulator decouples simulation from presentation.** Each frame accumulates the real (clamped) frame delta and runs the fixed schedule plus the registered fixed hooks a whole number of times at a fixed timestep, capped per frame so a stall cannot trigger an unbounded catch-up — the spiral-of-death guard. The variable schedule runs once per frame at display rate. The leftover accumulator becomes an interpolation alpha the engine surfaces through the `Time` resource; the engine performs no smoothing itself, leaving render-side interpolation to whatever consumes the alpha. This is the only model that keeps physics determinism (a fixed substep count) intact while letting rendering run free.

**The frontend and backend are split by extracted proxies, never a shared lock.** Render-relevant state crosses the frame boundary only as plain, backend-owned data — a camera proxy, the scene lighting, and a draw list of model-matrix-plus-resource-id entries — written by extraction routines at a defined sync point after the schedules run. A backend never reads live ECS storage while it renders, and the frontend never touches a buffer the backend is reading. The proxy store is double-buffered, so the lag between frontend and backend is structurally bounded to at most one frame regardless of configuration. Borrowing component storage across the pipeline, or guarding it with a lock that spans the frame, was rejected: the proxy discipline is what makes it impossible for the two halves to race.

**Low-latency sync is selectable from the first release, not retrofitted.** A `SyncMode` chooses whether the backend renders the proxies just extracted this frame (`Immediate`, zero frames in flight, the default) or the previous frame's (`Pipelined`, exactly one frame in flight). Pipelining trades latency for overlap; the immediate mode is always available rather than being a feature bolted on after unbounded pipelining became a latency problem.

**The render backend is a trait with a real and a headless implementation.** `WgpuBackend` owns a surface-owning renderer and a compiled lit render graph (a depth-only shadow pass feeding a lit forward pass) and resolves each draw proxy to its GPU mesh and material through the resource registry. `HeadlessBackend` validates and records the proxy buffer and presents nothing. The headless backend exists so the full frontend/backend split — extraction, double buffering, sync modes — runs in the headless example and the integration tests without a display server, and so an application can drive the loop frame by frame and inspect what was submitted.

**Physics is wired through an exclusive fixed hook, not a built-in system.** A physics step is `&mut World` exclusive work that runs at the fixed rate, so the engine exposes a fixed-hook registration (`FnMut(&mut World, &Time)`) rather than embedding physics directly. The application supplies the hook; the engine stays domain-agnostic and the physics crate stays a dependency only of applications that use it.

**`Time` is an engine-owned resource with a single writer.** spawn-core defines no time type, so the engine owns one, updates it at the top of each tick before any schedule runs, and mirrors it into the world resource store so systems read it through `Res<Time>`. The engine is its only writer; the resource is read-only to systems.

**Two drivers behind one tick.** `Engine::tick` runs exactly one platform-agnostic frame. The headless driver calls it in a deterministic loop with a fixed clock; the windowed driver calls it from the platform's redraw callback with a real-time clock. The windowed driver builds the engine inside the platform `init` callback, because the wgpu backend cannot exist until the window does, and stashes any build or tick error to surface from `run` after the event loop exits. Teardown follows field declaration order — render backend, input, audio, asset server, world — so the asset server's `Drop` joins its IO threads while the world that may still hold asset handles is alive.

## Architecture

The crate splits into focused modules, with the principal types re-exported at the crate root and a handful of downstream types (`Renderer`, `WindowConfig`, `Lighting`, and similar) re-exported for application convenience.

- **`app`** — `App`, the configuration aggregate: the world, the variable and fixed schedules organized by `ScheduleLabel` stages, a startup stage, fixed hooks, extraction routines, render-setup hooks, and the config. Builder methods register systems, events, resources, hooks, and extracts. `run` launches the windowed driver; `run_headless` and `build_headless`/`build_headless_with` produce a headless engine. The module also holds the `WindowedDriver` `PlatformApp` adapter.
- **`engine`** — the `Engine` runtime and the frame pipeline. `assemble` constructs the subsystems, inserts the `Time` resource, runs the startup stage once (without an event swap, so first-frame readers see startup events), then builds the variable and fixed schedules so startup-deferred resources and spawns are visible to them. `tick` runs the nine-step frame: clock advance, input begin-and-process, asset pump, the fixed-step loop, the variable schedule and event swap, proxy extraction, publish-and-render per sync mode, and the audio pump.
- **`config`** — `EngineConfig`: the fixed timestep, the per-frame fixed-step cap, the frame-delta clamp, the sync mode, and the window config, defaulting to a 60 Hz step with the immediate sync mode.
- **`frame`** — `SyncMode` (the double-buffer read policy) and `ScheduleLabel` (the ordered `First`/`Update`/`Last` stages of both schedules).
- **`render`** — the frontend/backend boundary: `CameraProxy`, `RenderProxy`, and the `RenderProxies` extraction buffer; the engine-private double-buffered `RenderProxyStore` that bounds frames-in-flight; the `RenderBackend` trait; and the `WgpuBackend` and `HeadlessBackend` implementations plus the `RenderSetup` hook type.
- **`time`** — the `Time` resource: frame and fixed-tick counters, real and simulated elapsed time, the per-frame delta, and the interpolation alpha, with engine-only mutators.
- **`error`** — `EngineError` (`#[non_exhaustive]`), composing the loop crates' error types with `From` conversions so `?` works across the pipeline, plus the `EngineResult<T>` alias.

## Constraints

- **Allocation.** Per-frame buffers are reused: the proxy draw list is cleared (not freed) each extraction and retains capacity, so a steady draw count is allocation-free. Hooks and extraction routines are boxed once at registration, never per frame. The one per-frame allocation in the wgpu backend is the resolved draw list, which borrows the resource registry and so cannot be a reused field without becoming self-referential — a deliberate, documented exception.
- **Safety.** 100% safe Rust, zero `unsafe`.
- **Panics.** No `unwrap`, `expect`, or `panic!` outside tests. Setup and frame failures return `EngineResult`; the windowed driver stores a frame error, requests exit, and surfaces it from `run` rather than aborting.
- **Determinism.** The headless driver advances by exactly the configured step each tick, so a headless run is reproducible; the engine preserves spawn-ecs's within-stage determinism and the fixed substep count that the physics determinism reference depends on.
- **Dependencies.** The eight loop subsystems — `spawn-core`, `spawn-ecs`, `spawn-platform`, `spawn-input`, `spawn-render`, `spawn-asset`, `spawn-audio`, `spawn-physics` — plus `std`. No third-party crates enter directly; winit reaches the build only transitively through `spawn-platform`.
- **Ownership and teardown.** The engine owns the world and every subsystem handle. Field declaration order is the teardown order, chosen so the asset server's IO threads join before the world holding asset handles drops.
- **Documentation.** Every public item carries a `///` contract doc comment; `#![deny(warnings)]` is in force at the crate root.

## Phase 2 Scope

In scope: the `App`/`Engine` aggregate; the nine-step frame pipeline; the fixed-step accumulator with the spiral-of-death cap and the surfaced interpolation alpha; the frontend/backend proxy split with the double-buffered store and the immediate/pipelined sync modes; the `RenderBackend` trait with the wgpu and headless implementations; the exclusive fixed-hook integration point; the windowed and headless drivers; the `Time` resource; the composed `EngineError`; and tests exercising the full loop to a clean shutdown headlessly.

Deferred, each to a later approved phase: multi-threaded render-command submission and a dedicated render thread (the split is in place, but submission is synchronous); richer schedule organization beyond the three ordered stages (system sets, run conditions, explicit inter-system ordering); asset hot-reload and a real asset-loading path wired into the loop (the engine constructs the asset server with hot-reload disabled because this phase loads no assets); and integration of scripting and networking into the frame (those subsystems are pumped by their own drivers and are not yet first-class loop steps). Interpolation between fixed ticks is surfaced as the alpha but not applied; render-side smoothing is left to the consumer. The line falls at a coherent, deterministic, headless-testable loop: everything that needs a real GPU surface, a thread pool, or a subsystem capability not yet wired waits for the phase where its prerequisite is real.
