# spawn-build

`spawn-build` is the asset build pipeline library at the base of the engine's tooling chain. It turns a directory of loose source files plus a small declarative manifest into a content-addressed output tree and a binary pack index that the runtime can mount. It exists to bridge authoring-time files and runtime-loadable assets while guaranteeing two properties the rest of the engine relies on: the same `AssetId` is derived for a path here as `spawn-asset` derives at load time, and identical inputs always yield byte-identical outputs so that builds are reproducible and incremental. Phase 1 ships only the library surface; the user-facing CLI wraps it later.

## Design Decisions

The dominant constraint is zero external dependencies. As the lowest crate in the tool chain, `spawn-build` links only `spawn-core` and `spawn-asset` and otherwise relies on `std` alone. Every facility that a third-party crate would normally supply is hand-rolled: recursive directory traversal over `std::fs::read_dir` instead of `walkdir`, a line-based `key=value` manifest parser instead of `toml`/`serde`, a hand-written FNV-1a hasher instead of a hashing crate, and `std::thread::scope` for parallelism instead of `rayon`. The payoff is a minimal, auditable trust surface and on-disk formats (manifest, cache) that diff cleanly in review and version control. The trade-off is a deliberately spartan feature set; a richer manifest schema remains a candidate for a future amendment.

Hashing uses FNV-1a 64-bit with fixed constants and a defined per-byte order. `std` offers no hasher guaranteed stable across runs, compiler versions, or platforms — `DefaultHasher` explicitly is not — so a self-contained algorithm with frozen constants is the only way to make content hashes portable and reproducible. FNV-1a fits the role: a dozen auditable lines, fast enough for change detection, with collision behavior acceptable because the hash drives change detection and content addressing only, never security. It is never to be trusted against adversarial collisions.

Determinism is treated as a correctness property rather than a nicety. Every collection that feeds serialized output is sorted by `AssetId` before writing; no timestamp, absolute path, hash-map iteration order, or thread-scheduling artifact is permitted to reach an output byte. This is what makes the parallel compile section safe — workers may finish in any order because all serialized data is re-sorted after the parallel region closes. Multi-byte integers in the pack index are written explicitly little-endian regardless of host byte order for the same reason.

`AssetId` identity is path-derived and content-independent, and the type itself is reused from `spawn-asset` rather than redefined. Editing a file's bytes changes its content hash but not its id. Sharing both the type and the exact FNV-1a-over-canonical-path algorithm with `spawn-asset` is the linchpin: the id baked into a pack entry equals the id the runtime computes for the same loose file, which is precisely how the runtime locates baked assets.

Robustness favors continuation over abort. A single unreadable or otherwise failing asset records its `BuildError` in the report and the build proceeds with the rest; a bad file never silently vanishes from the output, and it never poisons the parallel scope. I/O errors always carry the offending path and manifest/cache parse errors always carry a 1-based line number, so failures are actionable.

Several forward-looking hooks are defined without being exercised in Phase 1: the `HashMismatch` error variant reserves an integrity-verification step (re-hashing every skipped output each build would defeat the incremental fast path, so no verification runs yet), and `UnknownExtension` reserves a future strict extension policy. The active Phase 1 policy is permissive — every matched file is a copyable asset. Defining these now keeps later policy flips from introducing breaking error variants.

## Architecture

The crate is a flat set of single-responsibility modules under `src/`, with `lib.rs` carrying `#![deny(warnings)]`, declaring the modules, and re-exporting all public types at the crate root (including the re-exported `AssetId`). Modules stay public for explicit paths.

- `hash` — the `Fnv1a64` incremental hasher, its `FNV_OFFSET_BASIS_64`/`FNV_PRIME_64` constants, one-shot `hash_bytes`, streaming `hash_reader` (fixed-size chunks, bounded memory, default 64 KiB), and `canonical_relative_path`, which strips the source-root prefix, normalizes separators to `/`, and lexically resolves `.`/`..` without touching the filesystem, rejecting any path that escapes the root.
- `manifest` — `BuildManifest` and its `parse_file`/`parse_str` entry points. The parser splits each line on the first `=`, trims both sides, ignores blanks and `#` comments, and recognizes exactly `source_root`, `output_dir`, `include`, and `exclude`. `output_dir` is required; `source_root` defaults to the manifest directory; absent `include` defaults to a single `**`. Relative roots resolve to absolute paths against the manifest directory.
- `glob` — `Pattern`, with `compile`, `matches`, and `as_str`. The supported subset is frozen: `*` matches within a single path segment and never crosses `/`; `**` matches zero or more whole segments and is valid only as a complete segment, so a `**` fused to other characters is rejected at compile time. Matching is whole-path-anchored and case-sensitive.
- `discover` — `AssetEntry` (id, canonical source path, byte length, content hash) and the `discover` walk. It recurses with `read_dir`, skips the manifest file and a nested output directory, does not follow symlinks, applies include/exclude (exclude wins), hashes each included file via streaming, and returns the entries sorted by id. A duplicate `AssetId` is a hard error, never a silent overwrite.
- `cache` — `BuildCache` and `CacheRecord` over a tab-separated, id-sorted text file. `load` of a missing file yields an empty cache rather than an error. The incremental rule lives here: an asset is skipped only when a record's source hash matches the freshly discovered content hash and the expected output file is present on disk.
- `compile` — `compile_asset` and `CompileOutput`. The Phase 1 transform is the identity copy: source bytes go verbatim to `output_dir/data/<id-hex>`, written to a temporary sibling and renamed into place so a crash never leaves a half-written file that a later cache hit would trust. The output hash is computed from the written bytes independently, so it equals the source hash today but stays correct once real compilers replace the copy.
- `pack` — `PackEntry`, `PackIndex`, the format constants, and `write`/`read`. The format is a 16-byte little-endian header (`magic`, `version`, `entry_count`, zero `reserved`) followed by tightly packed entries in ascending id order, each carrying id, offset (always 0 in Phase 1), flags (`EXTERNAL` set), the length-prefixed UTF-8 canonical path, and the output content hash. `read` validates magic, version, and reserved fields and reports `PackFormat` on any mismatch or truncation.
- `pipeline` — `BuildPipeline`, `BuildConfig`, and `BuildReport`. The pipeline validates the manifest on construction (source root must exist as a directory; output directory need not), then `build` runs discovery, cache load, skip/compile partitioning, parallel compilation across a bounded worker pool, cache and index assembly from all discovered assets, and the two output writes. `clean` removes only `data/`, `index.spawnpack`, and the cache file within the output directory.
- `error` — `BuildError` (a `#[non_exhaustive]` enum) and the `BuildResult<T>` alias.

The high-level public shape: callers parse a `BuildManifest`, hand it to a `BuildPipeline` (optionally with a `BuildConfig`), and call `build` to get a `BuildReport` or `clean` to remove outputs. The hashing, glob, discovery, cache, compile, and pack types are exposed for direct use and testing but the pipeline is the intended front door.

## Constraints

- Allocation: discovery, cache, and pack collections allocate normally, but `hash_reader` streams through a caller-provided fixed-size buffer and never reads a whole file into memory, so arbitrarily large assets hash under bounded memory.
- Safety: no `unsafe` anywhere. No `unwrap()`, `expect()`, or `panic!()` outside test code — every fallible operation returns `BuildResult<T>`.
- Dependencies: only `spawn-core` and `spawn-asset`, both workspace crates. No other spawn crate and no external crate may appear in `Cargo.toml`; the implementation stays within `std`.
- Errors carry context: `Io` always includes the offending path; `ManifestParse` and `CacheParse` always include a 1-based line number. A bare `io::Error` without a path is rejected by design, so no blanket `From<std::io::Error>` exists — callers attach the path. `From<BuildError>` for `spawn_core::SpawnError` exists so build failures flow through the shared engine error type.
- Determinism is normative: identical source bytes plus an identical manifest produce a byte-identical `index.spawnpack` and byte-identical content-addressed outputs. No timestamp, absolute path, map iteration order, or thread schedule may leak into any output byte. Every serialized collection is id-sorted before writing.
- Path identity: paths recorded in entries and fed to hashing are canonical — relative to the source root, forward-slash normalized, lexically resolved — matching `spawn-asset`'s canonical-path contract. The `AssetId` algorithm is shared verbatim with `spawn-asset`.
- Endianness: every multi-byte integer in `index.spawnpack` is written explicitly little-endian regardless of host endianness.
- A failing asset never disappears silently: its error lands in `BuildReport.failed` and the build continues. A duplicate `AssetId` is always a hard error, never an overwrite.
- All public items carry `///` documentation describing their contract.

## Phase 1 Scope

In scope: the line-based manifest and its hand parser, deterministic filesystem discovery over the frozen `*`/`**` glob subset, the shared content-independent `AssetId`, FNV-1a 64-bit content hashing, an incremental cache that skips unchanged assets, the identity compile step producing content-addressed `data/<id-hex>` files, the byte-precise `index.spawnpack` writer and reader, the `BuildPipeline` orchestrator with bounded parallel compilation, the `BuildReport`, the `BuildError`/`BuildResult` layer, and unit plus integration tests covering all of it.

Explicitly deferred, each pending its own approved amendment:

- The `spawn` CLI binary that wraps `BuildPipeline` — a later phase. Phase 1 exposes library APIs only.
- Real per-format asset compilers for images, meshes, audio, and shaders. These arrive with their owning crates' later phases and register against this pipeline then; the identity step is their explicit predecessor, and the extension-to-loader registry is recorded as the intended integration point without being consulted yet.
- Single-blob packing — concatenating outputs into one file with real non-zero offsets. Phase 1 stores each asset as a separate external file, which is why every pack entry carries offset 0 and the `EXTERNAL` flag.
- Compression, encryption, asset dependency graphs, a richer (toml/serde/binary) manifest schema, watch-mode and continuous builds, remote and distributed build caches, and an integrity verification pass (the `HashMismatch` variant is reserved for it).
- Platform packaging, installers, code signing, and console targets.

The line was drawn to deliver a complete, reproducible, incremental copy-through pipeline with a stable on-disk contract while everything that adds format knowledge, output layout complexity, or external surface waits for a dedicated phase. The identity transform and the external-storage pack flag are the seams chosen so later phases can replace compilation and packing without disturbing any other contract.
