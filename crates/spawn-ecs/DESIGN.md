# spawn-ecs

`spawn-ecs` is the entity-component-system core of the engine: it owns entity identity, component storage, queries over that storage, and the scheduler that drives systems each frame. Every higher-level crate builds on the data model and execution model defined here: rendering, physics, scripting, editor, networking, UI, and debug tooling. The crate is domain-agnostic; it stores arbitrary component types and knows nothing about game semantics. The single goal is a cache-friendly, deterministic, parallel data store with zero per-frame allocation in steady state.

## Design Decisions

**Archetype storage over sparse sets.** Entities are grouped into archetypes, where an archetype is a unique sorted set of component types. Each archetype stores its components as Structure-of-Arrays columns (`Vec<T>`-equivalent per component) alongside a parallel `Vec<Entity>`. All entities in an archetype share an identical memory layout, so iterating a query is a linear scan over contiguous columns, the cache-friendly path that data-oriented systems care about. Sparse-set storage, which trades iteration locality for cheaper insert/remove, is deliberately excluded; the workloads that dominate a game loop are reads and iteration, not structural churn.

**Explicit component registration, no blanket impl.** The `Component` trait is a marker with no methods, but a blanket `impl<T> Component for T` is rejected. Each component type opts in with an explicit `impl Component for T {}`. This keeps the set of component types auditable and prevents accidental registration of arbitrary types: the full list of what can live in the world is grep-able. The only engine-supplied impls are `Transform3D` and `Transform2D` from `spawn-core`, which exist here purely because the orphan rule forces the impls into the trait-owning crate; they are first-class components consumed by physics, scripting, and the editor.

**Deferred structural mutation through `Commands`.** Systems never receive `&mut World`. They read through queries and express structural changes (spawn, despawn, insert, remove) by recording them into a per-system command buffer. Buffers are drained at the stage boundary against `&mut World`. This is the keystone that makes intra-stage parallelism sound: during execution every system holds only a shared `&World` plus its own private buffer, so no two systems can race on storage. It also makes ordering explicit and reproducible rather than dependent on interleaving.

**Determinism is a hard contract, not a best effort.** System registration order defines a total order. That order drives conflict-batch assignment and command-buffer application order. Parallel execution is constrained so that observable post-stage world state is byte-identical regardless of thread scheduling, and identical between single-threaded and multi-threaded runs. Nondeterminism in results would be a bug, tested for directly under a stress loop.

**`std::thread::scope` for parallelism, no thread pool.** Parallelism within a batch uses scoped threads from `std` only. No `rayon`, no global worker pool. Scoped threads borrow stack data without `'static` gymnastics and keep the dependency surface at zero. A persistent-worker design is a possible future direction if scope-thread spawn cost ever proves to dominate, but that is a separate effort with its own justification.

**Zero `unsafe`.** The entire crate is safe Rust. Type-erased component storage, clone and drop over erased columns, is built from safe `std` function pointers over `Vec<T>`, not raw pointer manipulation. Should a future profiling pass prove `unsafe` unavoidable for column access, that change carries a mandatory `SAFETY` justification and stands on its own; the baseline ships with none.

## Architecture

The crate is split into focused modules, with the public API re-exported at crate root (`spawn_ecs::Entity`, `spawn_ecs::World`, `spawn_ecs::Query`, and so on); modules stay public for explicit paths.

- **`entity`**: The `Entity` handle (`#[repr(C)]`, 8 bytes: a `u32` index plus a `u32` generation), the slot allocator, and generation recycling. `Entity` is opaque with no public constructor; only `World` mints them. A `PLACEHOLDER` sentinel exists so downstream code can default-initialize entity fields without wrapping in `Option`.
- **`component`**: The `Component` marker trait, the dense `ComponentId` (`#[repr(transparent)]` `u32` assigned in registration order from zero), and the registry that records each type's `TypeId`, size, alignment, and erased clone/drop vtable. Registration is idempotent.
- **`world`**: `World` ties everything together. Public surface: structural ops (`spawn`, `spawn_with`, `despawn`, `insert`, `remove`, `contains`, `has`), lookup (`get`, `get_mut`, `entity_count`), and constructors for queries and commands (`query`, `query_mut`, `commands`, `apply_commands`). `World` is `Send` and not `Clone`; intra-stage parallel access is mediated by disjoint borrows, never by sharing `&mut World`.
- **`archetype`**: Column storage internals. Almost entirely private; the only public surface is `ArchetypeId` (opaque, for debug/metrics) and `World::archetype_count`. Insert/remove are implemented as swap-remove-plus-push moves between archetype columns.
- **`bundle`**: The sealed `Bundle` trait, implemented for single components and for tuples up to arity 12. A bundle names a component-id set and supplies the erased writes that place a new entity into the correct archetype in a single move.
- **`query`**: `Query`, the sealed `QueryData` trait (`&T`, `&mut T`, `Entity`, and tuples to arity 12), and the iterators. `query/filter.rs` holds the sealed `QueryFilter` trait with `With<T>` and `Without<T>`. Queries iterate by zipping matching archetype columns; items borrow directly into storage with no intermediate buffer. `QueryIter`/`QueryIterMut` are `Iterator` + `ExactSizeIterator`. Random access (`get`/`get_mut`) is `O(1)` through the entity-to-location map plus a filter check.
- **`system`**: The `System` trait (`name`, `access`, `run` over `&World` and `&mut Commands`), the `IntoSystem` function adapter, and access tracking. `Access` holds a read mask and a write mask, each a `ComponentMask`, a growable `Vec<u64>` bitset keyed by `ComponentId`. The adapter computes a system's access by unioning its queries' read/write sets at construction. Masks expose read-only `contains`, `intersects`, and `is_empty`; they have no public constructors or mutators.
- **`commands`**: `Commands` and the command buffer. Records deferred structural mutations during `run`; buffers are allocated once at build time and cleared (not freed) each frame. `Commands::spawn` returns a reserved entity id immediately, drawn from an atomic high-water counter so concurrent systems never collide.
- **`schedule`**: `Stage` and `Schedule`. A stage holds ordered systems, a conflict graph, and parallel batches; a schedule holds ordered stages. `Stage::build` resolves access sets and partitions systems into batches; `Schedule::run` executes batches in order and applies command buffers at each stage boundary.
- **`error`**: `EcsError` (`#[non_exhaustive]`) and the `EcsResult<T>` alias.

## Constraints

- **Allocation.** No heap allocation in `Schedule::run` after the first call (warm-up). Allocation is permitted only during archetype creation, component registration, archetype moves triggered by insert/remove/spawn into a never-seen archetype, and command-buffer growth. Query iteration is strictly allocation-free: no item boxing, no per-iteration match vector. Once the archetype graph is warm, moves between existing archetypes reuse columns (push/swap-remove) with amortized no allocation. A test wraps a second `run` in a counting global allocator and asserts zero allocations.
- **Safety.** 100% safe Rust. No `unsafe` anywhere in the baseline. Any future `unsafe` requires a separate change and a mandatory `SAFETY` comment.
- **No panics in non-test code.** No `unwrap`, `expect`, or `panic!` outside tests. Fallible lookups return `Option`; fallible operations return `EcsResult`. Out-of-bounds access and stale-identity mismatches resolve to `None` or `EcsError`, never a panic. A panicking system is caught at the scoped-thread join and surfaced as `EcsError::SystemPanicked`, not a process abort.
- **Dependencies.** `spawn-core` and `std` only. No third-party crates, verifiable through `cargo tree -p spawn-ecs`. This crate is depended on by render, physics, script, editor, net, ui, and debug.
- **Component invariants.** Components are plain data: `Send + Sync + 'static`, no interior mutability, no hidden state. The `Send + Sync + 'static` bound is what makes parallel scheduling sound. `Copy`-ness is a review convention rather than a trait bound, since some legitimate components (e.g. fixed-capacity arrays) are `Clone`-only; storage requires `Clone` at minimum via the registration vtable. Systems are stateless functions over queries and hold nothing between runs.
- **Generation safety.** A stale `Entity` never aliases a live one. Generations bump on recycle and saturate at `u32::MAX`; a saturated slot is retired permanently to preserve the no-aliasing guarantee, leaking one `u32` slot as an accepted cost. Any operation handed an entity whose generation does not match the live slot treats it as not-live.
- **Conflict relation.** Two systems conflict iff their writes intersect, or one's writes intersect the other's reads. Read-read sharing never conflicts. Command writes are deferred and never enter the conflict relation, since they touch storage only at the stage boundary.
- **Documentation.** Every public item carries a `///` doc comment expressing its contract.

## Phase 1 Scope

In scope: entity identity and recycling; archetype SoA storage; explicit component registration; the `World` mutation and lookup API; the `Query` engine with `With`/`Without` filtering and allocation-free iteration; the `System` abstraction with read/write access sets and conflict detection; the `Schedule`/`Stage` runner with deterministic ordering and `std::thread::scope` parallelism; deferred structural changes via `Commands`; the `EcsError` type; and unit tests covering all of the above.

Explicitly deferred, each gated behind its own future approval: change detection and `Changed<T>`/`Added<T>` filters; resource storage (`Res`); events; entity relationships and hierarchies; system parameters beyond `Query` and `Commands`; sparse-set storage; component hooks and observers; serialization; reflection; async systems; any thread-pool crate such as `rayon`; dynamic component registration from external data; and cross-frame query caching. The `AccessConflict` error variant is reserved for a future opt-in strict-build mode and is never returned by the default `build`, which resolves conflicts by serializing systems into separate batches rather than erroring.

The line is drawn at the minimum coherent storage-and-scheduling core. Everything deferred either composes on top of this foundation without changing it (resources, events, change detection) or represents a distinct storage or execution strategy whose tradeoffs deserve their own evaluation (sparse sets, persistent worker pools, async). Shipping the core first keeps the surface small, fully tested, and stable for the seven crates that depend on it.
