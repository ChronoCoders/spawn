# spawn-replication

`spawn-replication` is the server-authoritative replication layer that sits above the byte transport. It decides what state each client should hold and keeps every client converged on the server's authoritative world through interest-filtered, delta-compressed snapshots, while routing client input and ownership-gated remote calls back to the server. It is a library, pumped once per fixed tick like the transport beneath it — no threads, no async runtime, no internal clock. The division of labor is deliberate: this crate owns mechanism — the replicated-component registry, the relevancy markers, interest management, the serialization bridge, the snapshot and delta machinery, role bookkeeping, and the composition of the transport — while the game owns policy: which components replicate, what remote calls mean and how to validate them, which connection owns which entity, and the relevancy parameters. The server holds the one true state; clients send input and ownership-gated calls and otherwise render what the server tells them.

## Design Decisions

**Mechanism here, policy in the game.** The engine supplies marker components, a registration mechanism, interest-management machinery with hook points, and the snapshot/transport pipeline; the game declares which components replicate, defines and validates remote calls, assigns ownership, and sets the relevancy policy. This split keeps the replication layer reusable across games and keeps gameplay rules where they belong. It is realized as a new crate above the transport rather than a module inside it, because the transport's contract is to move bytes without interpreting them — replication is the layer that interprets.

**Relevancy class is a marker component; per-client visibility is a server resource.** Four marker components express an entity's relevancy class — replicated at all, always relevant, owner-only, or static. In an archetype world a marker places an entity in a distinct archetype, so the set of replicable entities is a small set of archetypes to iterate rather than a full-world scan. The per-client visibility matrix, by contrast, lives in a single server-side resource as compact current/previous bitsets indexed by a dense entity id — never as components on entities. Storing per-client state on entities was rejected: it scales the storage by client count per entity and churns archetypes as visibility changes, where the resource-side bitset matrix is far smaller and leaves archetypes stable.

**Roles are derived, not stored.** An entity's network role — authority on the server, an autonomous proxy on its owning client, a simulated proxy elsewhere — is computed per peer from ownership and locality, never written onto the entity. The role is a pure function of who owns the entity and where the question is asked.

**A dense, recycled entity id keeps the wire handles and the bitsets compact.** Each replicated entity is assigned a dense `ReplId` that is the on-wire handle and the bitset index; freed ids return to a free list and are reused, so the id space tracks the peak number of concurrently live replicated entities rather than the total ever spawned. This keeps the per-client visibility bitsets small.

**Components serialize through a fn-pointer registry.** A component opts into replication by implementing a trait built on the bit codec; registration records a fn-pointer vtable that drives the component's single serialization function in either direction, with a dense wire id assigned by registration order. There is no per-field dynamic dispatch beyond one type-erased stream indirection, and no per-access allocation — the same discipline the ECS reflection layer uses. Both peers must register the same components in the same order; a build-time manifest hash makes a mismatch a detectable rejection rather than a silent desync.

**Interest management is the validated layered model.** Static buckets — always-relevant, owner-only, and placed-once static entities — are cheap and handled first; everything else flows through one dynamic spatial filter, a sparse spatial-hash grid with an incremental per-entity cell cache. Each tick the grid is synced for entities that changed cells, then per client a block of cells around its viewer is gathered and the resulting visibility set is diffed against the previous tick to emit exactly-once spawn and despawn deltas. The default cell size is half the view radius, and the gather block is sized to the **exit** radius rather than the enter radius: a twenty-five-percent exit-radius hysteresis keeps a previously-visible entity visible past the enter boundary, so the block must reach that far or a band entity would fall outside it and be spuriously despawned — the exact thrash the hysteresis exists to remove.

**Snapshots delta against the client's acked baseline.** Each tick the server sends a client the visibility delta plus a state delta for visible entities, delta-compressed against the most recent snapshot the client has acknowledged. The delta is per serialization unit — a one-bit changed marker per field, with the value sent only when it differs from the baseline, and a no-baseline encoding that is simply absolute. Both peers keep a ring of recent per-tick states; the states agree because they are absolute serializations of the same values, so a changed field on one side decodes correctly on the other only if the baseline cursor advances in lockstep, which it does. A per-packet byte budget bounds each snapshot; when the full delta will not fit, updates are ordered by a staleness priority accumulator and packed until the budget is reached, with the truncated entities rising in priority. A baseline too large for one datagram fragments through the transport.

**Simulated proxies interpolate in the past, into a separate component.** Remote entities render behind real time by a delay (two snapshot intervals by default) so two snapshots always bracket the render time, with Hermite position interpolation using a transmitted velocity and slerp orientation. The interpolated transform is written to its own component; the authoritative transform is never overwritten by interpolation, so the two never fight.

**Prediction is opt-in, and ships snap-only.** A client may mark its owned entity predicted; it then tags inputs with a sequence, applies them locally, retains the unacked ones, and on each authoritative snapshot snaps to the server state. Replaying the unacked inputs on top requires a deterministic fixed-tick simulation, which in turn leans on ECS change detection that does not yet exist, so snap-only reconciliation is the shipping baseline and a replay helper is provided for when that prerequisite lands. The visual error correction — snap a large divergence, smooth a small one with a decaying offset — applies in both modes.

**Remote calls are gated by ownership, validated by the game.** The Server, Client, and Multicast call directions travel over the reliable-ordered channel. A client may only invoke a Server call on an entity it owns — the ownership gate is the first line of anti-cheat, enforced before any handler runs — and the game validates the payload, because the engine cannot know game rules. A one-byte message-kind tag multiplexes snapshots, input packets, and remote calls over the transport's channels without the transport learning anything about replication.

## Architecture

The crate is a flat set of modules with the public types re-exported at the crate root.

- **`id`** — `ReplId` and `ReplIdMap`: the dense, free-list-recycled entity id and the bidirectional entity↔id map, with server allocation and client binding paths.
- **`markers`** — the `Replicated`, `AlwaysRelevant`, `OwnerOnly`, and `StaticRelevant` component types.
- **`role`** — the `NetRole` triad and its per-peer derivation from ownership and locality.
- **`registry`** — the `Replicate` opt-in trait and the `ReplicationRegistry` of fn-pointer vtables, with dense wire ids and the manifest hash.
- **`interest`** — `ReplicationVisibility`, the server resource holding the spatial grid and the per-client current/previous bitsets, the gather-then-diff loop, and the spawn/despawn deltas, with `VisibilityConfig` for the cell-size and hysteresis factors.
- **`snapshot`** — the delta-or-absolute stream adapters, `SnapshotState` (a tick's recorded absolute component bytes), the snapshot encode and decode over a baseline, the header peek, and the rate/budget/history constants.
- **`interp`** — the `InterpBuffer` sample ring and the `InterpolatedTransform` render component.
- **`predict`** — the `Predicted` marker, the `InputBuffer`, the `replay` helper, and the snap-versus-smooth `PredictionSmoother`.
- **`rpc`** — `RpcKind`/`RpcId`/`RpcHeader`, the `RpcRegistry`, the encode/decode helpers, and the `server_rpc_authorized` ownership gate.
- **`server`** — the `Replicator` driver: per-client snapshot rings and baseline acks, the id-sync and interest passes, snapshot construction with the priority accumulator, and inbound input/RPC routing, surfaced as `ServerEvents`.
- **`client`** — the `ReplicationClient` driver: snapshot application against the baseline ring, the input packet with the snapshot ack and the redundant input window, and inbound RPC routing, surfaced as `ClientEvents`.
- **`config`** — `ReplicationConfig` (send budget, default view radius); **`wire`** — the message-kind tags; **`error`** — `ReplError` (`#[non_exhaustive]`) and `ReplResult<T>`.

## Constraints

- **Allocation.** Per-client bitsets, the spatial grid, the snapshot rings, and the encode/decode scratch are preallocated and reused; growth is bounded by client count and the peak replicated-entity count. The snapshot/baseline-recording path currently allocates per component per tick (the recorded absolute bytes and the per-client delta vectors); this is a known deviation from the zero-steady-state-allocation target, recorded for a later hoist into pooled scratch.
- **Safety.** 100% safe Rust, zero `unsafe`. Per-component dispatch is fn-pointer vtables, not transmute or raw pointers.
- **Panics.** No `unwrap`, `expect`, or `panic!` outside tests. A malformed snapshot or remote call, an unknown component wire id, a stale baseline, or a missing entity is an error or a no-op, never a panic — a server must not crash on a hostile client and a client must not crash on a malformed server packet. A spawn id beyond a fixed cap is rejected so a hostile snapshot cannot drive an unbounded client-side allocation.
- **Determinism.** Snapshot encoding is bit-exact and platform-independent through the bit codec; the gather-then-diff visibility computation is deterministic given the world.
- **Authority.** Clients never originate authoritative state; the only client-to-server traffic is input and ownership-gated, game-validated remote calls.
- **Transport boundary.** The crate never touches a socket directly — it composes the transport's server and client and their channels, and the transport remains the sole owner of the UDP socket.
- **Dependencies.** `spawn-core`, `spawn-ecs`, `spawn-net`, `spawn-serialize`, and `std` only. No third-party crates, no `serde`, no async runtime — verifiable through `cargo tree -p spawn-replication`.
- **Documentation.** Every public item carries a `///` contract doc comment; `#![deny(warnings)]` is in force at the crate root.

## Phase 2 Scope

In scope: the role triad and the four relevancy markers; the `Replicate` trait and the fn-pointer registry with the manifest hash; the dense recycled `ReplId`; interest management with the spatial-hash grid, the per-client visibility bitsets, the gather-then-diff loop, and the exit-radius gather block with hysteresis; acked-baseline delta-compressed snapshots with the send budget and the staleness priority accumulator; snapshot interpolation into a separate component; opt-in prediction with snap-only reconciliation and the visual error correction; the Server/Client/Multicast remote-call taxonomy with the ownership gate; the message-kind framing; the `Replicator` and `ReplicationClient` drivers; and a headless loopback test proving two clients converge on the server world under interest filtering, including a baseline that exceeds the datagram limit and so exercises fragmentation.

Deferred, each to a later approved phase: prediction replay reconciliation, gated on ECS change detection (snap-only ships now); component-level dirty-tracking deltas, which also need change detection; encryption, NAT traversal, matchmaking, congestion control beyond the per-tick budget, lag compensation (server-side rewind for hit detection), and authority transfer; line-of-sight or frustum relevancy refinement layered on the spatial grid; a derive macro for the `Replicate` trait; the hoist of the snapshot/baseline path off the per-tick allocation; and auto-wiring of typed remote-call dispatch and interpolation feeding, which the drivers leave to the game by surfacing raw bytes and the interpolation buffer. The deterministic-rollback foundation is the other, mutually exclusive networking fork and is explicitly not built under the recorded server-authoritative decision. The line falls at a complete, headless-testable server-authoritative pipeline whose remaining pieces depend on capabilities — change detection, real congestion control, a security layer — that warrant their own phase.
