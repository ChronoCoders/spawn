# spawn-editor

`spawn-editor` is the headless framework that backs the visual scene editor: a data model and command infrastructure that a later-phase editor shell drives. It carries no rendering, no window, no widgets, and no input handling. The engine and its editor are built together rather than retrofitted, and this crate is the foundational layer of that effort — the undo/redo engine, selection model, edit/play state machine, and gizmo geometry math that every higher-level editor feature will sit on top of. It operates on a `spawn_ecs::World` borrowed from its caller and owns no scene data of its own beyond the revert state captured inside individual commands.

## Design Decisions

The command pattern is the spine of all mutation. Every scene edit travels through a `Command` trait object with paired `apply`/`revert` methods, and a bounded `CommandStack` orchestrates execution, undo, redo, history capacity, and dirty tracking. The governing correctness property is round-trip identity: an `apply` followed by its `revert` leaves the `World` observably unchanged. This is favored over an event-sourcing or full-snapshot-per-edit scheme because a per-command revert pays only for the state it actually mutates, keeps undo/redo at O(1), and never requires diffing the world.

Command revert state lives inside each command rather than in a centralized store. A command captures whatever it needs to undo itself — typically the prior value of a component it overwrites — at the moment of first `apply`. This latch-on-apply design is the reason `apply` takes `&mut self`. Centralizing revert state was rejected: it would couple the stack to the concrete shape of every command's captured data and reintroduce the reflection problem the ECS does not solve.

Component handling is deliberately explicit, not generic. spawn-ecs has no reflection or serialization, so no command can blindly snapshot an arbitrary entity. Each command instead names the concrete component types it touches and captures them by value through `World::get`/`insert`/`remove`. The editor-managed component set is exactly `Transform3D` for now; widening it is a deliberate, reviewed change that grows the capture structs in lockstep. A type-erased blob approach is impossible against the current ECS surface and was not pretended otherwise.

Gesture coalescing is handled by command merging rather than by stack-level batching. A continuous drag emits many `SetTransform3D` per frame; `try_merge` lets the top-of-stack command absorb a successor so the whole gesture collapses into one undoable step. The absorbing command adopts the successor's target value while keeping its own original captured revert state, so a single undo rewinds the entire drag. Merge eligibility is the command's own decision; the stack only offers merges for consecutive edits with no interposed command and no crossed redo boundary.

Heap-allocated `Box<dyn Command>` trait objects are accepted without apology. Command construction and stack mutation occur on the editor interaction path — a user gesture — never inside the per-frame engine schedule, so they fall outside the allocation-free budget that governs hot-path code. This is stated plainly so the allocation is not mistaken for an oversight during review.

The play-mode snapshot ships as a contract now with a scoped implementation. The full generic world snapshot — every component, byte-exact, with stable entity ids — depends on spawn-ecs gaining world cloning or serialization, which does not exist yet. Rather than block the mode state machine on that, the `enter_play`/`exit_play` API and the Edit/Play transitions land complete, with snapshot/restore implemented over the managed component set (`Transform3D` plus the live-entity roster) using the same per-known-type capture as `DespawnEntity`. The edges this leaves — non-`Transform3D` components added during play survive the restore, and entities respawned on restore come back with `Transform3D` only — are documented on the public API so the limitation is visible rather than silent.

Selection is backed by an ordered `Vec`, not a `HashSet`. Selection sets are editor-scale, not engine-scale, so linear membership checks are cheap and insertion order gives stable UI listings for free. Selection change notification is pull-based: there is no event bus in Phase 1 spawn-ecs, so any mutator that actually changes state returns `true`, and the caller turns that into a `SelectionChanged` snapshot for the UI layer. No-op mutations return `false` and emit nothing.

Errors are explicit and never abort the process. A corrupt undo stack producing a failing `apply` or `revert` returns `Err` and leaves the stack intact; it must not panic. `EditorError` is `#[non_exhaustive]` and avoids `Clone`/`PartialEq` for forward compatibility and because it wraps a non-`Clone` ECS error. Static string slices for context keep error construction allocation-free, matching spawn-core convention.

## Architecture

The crate splits into seven modules, with `lib.rs` declaring them, applying `#![deny(warnings)]`, and re-exporting the principal types at the crate root while leaving the modules public for explicit paths.

`error` defines `EditorError` and the `EditorResult<T>` alias. `EditorError` covers entity-not-found, missing-component, nothing-to-undo/redo, invalid-mode, and a wrapped ECS error variant; it implements `Error` (with `source()` surfacing the inner ECS error), `Display`, and `From<EcsError>`.

`command` holds the `Command` trait: `apply`, `revert`, `label`, an `as_any` accessor for downcasting, and a `try_merge` with a default `false` implementation. The trait is `'static` so commands can be boxed and stored. `try_merge` implementations downcast a candidate successor via `as_any().downcast_ref::<Self>()` and bail to `false` on a mismatch.

`command::builtin` provides the three Phase 1 commands. `SpawnEntity` creates a bare or transform-bearing entity, records the resulting `Entity` for revert and exposure, and despawns it on revert; a redo spawns a fresh entity whose id may differ, so callers read the current id through `entity()` rather than caching it. `DespawnEntity` captures the entity's managed components into a fixed `Option`-per-type struct before despawning, and on revert respawns and reinserts what it captured — producing a new `Entity` id that the command then tracks. `SetTransform3D` is the mergeable one: it latches the prior transform on first apply, writes the new value, reverts to the captured value, and merges with another `SetTransform3D` targeting the same entity.

`stack` contains `CommandStack`: undo and redo `Vec<Box<dyn Command>>`, a configurable capacity, a save-point cursor, and the merge plumbing. Its surface covers construction (`new`, `with_default_capacity`), introspection (`len`, `is_empty`, `capacity`, `can_undo`/`can_redo`, `undo_label`/`redo_label`), execution (`execute`, `execute_merged`), history navigation (`undo`, `redo`, `clear`), and dirty tracking (`mark_saved`, `is_dirty`). Execution applies before pushing and clears the redo stack on any new edit; capacity eviction removes the oldest entry and adjusts the save-point bookkeeping. A failed `apply` propagates the error and leaves the stack untouched.

`selection` defines `Selection` — an ordered entity set with a distinguished primary — and the `SelectionChanged` snapshot struct. Mutators (`select`, `deselect`, `toggle`, `clear`, `set_primary`) return whether they changed state; queries cover `primary`, `is_selected`, `is_empty`, `len`, and ordered `iter`. `retain_live` drops entities no longer present in a borrowed `World`, which reconciles selection after despawn undo/redo. `snapshot` produces the `SelectionChanged` the caller forwards to the UI.

`state` aggregates a `CommandStack` and a `Selection` into `EditorState`, plus the `EditorMode` enum and a private play-snapshot slot. `enter_play` captures the managed snapshot and transitions Edit to Play; `exit_play` restores it, transitions back, and calls `retain_live` to drop entities the restore invalidated. Both reject the wrong starting mode with an invalid-mode error.

`gizmo` is pure math over spawn-core types with no state, no `World`, and no rendering. It provides a `Ray`, an `Axis` enum with `unit`, and closed-form functions: ray-against-axis-handle hit testing, axis and plane drag deltas, ray/plane intersection, and signed rotation angle about an axis. Every function is total — degeneracies return a documented sentinel (`None`, `0.0`, or `Vec3::ZERO`) rather than panicking — and none allocate.

## Constraints

- Allocation: command boxing and history `Vec` growth are permitted because they live on the editor interaction path, outside the engine's per-frame allocation-free budget. Gizmo math allocates nothing. Selection operations run over a small `Vec` with no auxiliary allocation. Error construction is allocation-free via `&'static str` context.
- Safety: 100% safe Rust. No `unsafe` anywhere in the crate.
- Errors: no `unwrap`, `expect`, or `panic!` outside test code. Fallible operations return `EditorResult`; genuine absence returns `Option`. A failing `apply` or `revert` returns `Err` and never aborts.
- Dependencies: `spawn-core`, `spawn-ecs`, and `std` only. No third-party crates, verifiable through `cargo tree`. No dependency on `spawn-render`, `spawn-ui`, `spawn-input`, or `spawn-platform`.
- World ownership: the crate borrows a `World` and never owns one. The only scene state it retains is the revert data captured inside live commands.
- Component awareness: every command names the concrete component types it manipulates. The managed set is exactly `Transform3D`; expanding it grows the relevant capture structs in the same reviewed change.
- Conventions: numeric and coordinate conventions follow spawn-core — right-handed, radians, column-major. Gizmo math honors them, and float comparisons use spawn-core `EPSILON`/`ApproxEq` rather than exact equality.
- Documentation: every public item carries a `///` contract doc comment, since rustdoc is the API contract.

## Phase 1 Scope

In scope: the `Command` trait with apply/revert/merge; the bounded `CommandStack` with undo/redo, capacity eviction, and dirty/save-point tracking; the three built-in commands (`SpawnEntity`, `DespawnEntity`, `SetTransform3D`); the `Selection` model with a primary and change snapshots; the `EditorState` aggregate with the Edit/Play state machine and the play snapshot/restore contract; the pure gizmo math library; the `EditorError`/`EditorResult` types; and exhaustive unit tests for all of it.

Deferred to a later approved phase: scene-view rendering, inspector and property UI, gizmo drawing, picking against rendered geometry, viewport and camera controllers, the asset browser, scene serialization to and from disk, copy/paste, multi-document and tab management, prefabs, and entity hierarchy/parenting. The visual editor shell that consumes this framework integrates later. Generic full-world snapshot — every component, byte-exact, with stable ids — is gated on spawn-ecs gaining world cloning or serialization and is out of scope until that lands.

The line falls between the editor's data model and its presentation. Everything that can be specified, built, and tested against the current ECS surface without a window or a renderer lands now; everything that needs pixels, picking, or an ECS capability that does not yet exist waits for a phase where its prerequisites are real. The play-snapshot boundary sits exactly where spawn-ecs runs out of capability: the API and state machine are complete, the implementation reaches as far as the managed component set allows, and the gap is documented rather than faked.
